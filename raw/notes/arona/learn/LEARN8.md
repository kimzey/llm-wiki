# 🔍 เรียนรู้ Arona (ตอนที่ 8): การค้นหาข้อมูล (The Library - Vector Search & Retrieval)

หัวใจของระบบ RAG คือ "การหาข้อมูลที่แม่นยำ" ในหน้านี้เราจะมาดูเทคนิคที่ Arona ใช้เพื่อค้นหาเอกสารที่ตรงใจ User ที่สุดครับ

---

## 1. การค้นหาแบบ Hybrid Search
Arona ไม่ได้ค้นหาแค่แบบเดียว แต่ใช้ 2 วิธีควบคู่กัน (อยู่ใน `src/modules/ai/const.ts`):

### ก. การค้นหาแบบเวกเตอร์ (Vector Search)
*   ใช้ **`pgvector`** ในการหา "ความหมาย" ของประโยค
*   **จุดเด่น:** หาเจอแม้คำจะไม่ตรงกันเป๊ะๆ เช่น ค้นหา "วิธีลางาน" จะเจอข้อความที่พูดถึง "ระเบียบการหยุดพักผ่อน"
*   **สูตรลับ:** Arona ผสมผสานน้ำหนักระหว่าง `title_embedding` (15%) และ `content_embedding` (85%) เพื่อให้ได้ผลลัพธ์ที่สมดุลที่สุด

### ข. การค้นหาแบบข้อความ (BM25 Full-Text Search)
*   ใช้ **`ParadeDB`** ในการค้นหา "คำหลัก" (Keywords)
*   **จุดเด่น:** หาเจอแม่นยำมากเมื่อมีคำเฉพาะ เช่น ชื่อโปรเจกต์ "Arona" หรือรหัสสินค้า
*   **สูตรลับ:** มีการทำ Normalization คะแนนเพื่อให้เปรียบเทียบกับผลลัพธ์จาก Vector ได้ง่ายขึ้น

---

## 2. ขั้นตอนการทำงานของ `search()`
เมื่อ User ถามมา ระบบจะทำขั้นตอนดังนี้:
1.  **BM25 Search:** วิ่งไปหาจากคำหลักก่อน (ถ้าเจอเยอะพออาจจะจบที่นี่)
2.  **Vector Search:** ถ้ายังต้องการเพิ่ม หรือหาจากคำหลักไม่เจอ จะใช้การแปลงคำถามเป็นเวกเตอร์แล้ววิ่งไปหาในอวกาศ (Vector Space)
3.  **Deduplication:** นำผลลัพธ์มารวมกัน และตัดตัวที่ซ้ำออก เพื่อส่งให้ AI อ่าน

---

## 3. การกรองข้อมูล (Permission Filtering)
นี่คือจุดที่ Arona แตกต่างจากระบบอื่น:
*   ทุก Query จะมีเงื่อนไข `WHERE` เสมอ เพื่อเช็ค `access_level` และ `department`
*   **Security by Design:** ไม่มีการดึงข้อมูลออกมาก่อนแล้วค่อยกรอง แต่กรองตั้งแต่ในระดับฐานข้อมูล (SQL Level) เพื่อความปลอดภัยและความเร็วสูงสุด

---

## 4. ทำไมต้องใช้ ParadeDB?
เราเลือกใช้ ParadeDB (ที่สร้างบน PostgreSQL) เพราะ:
1.  **All-in-One:** เก็บข้อมูลธรรมดา, เวกเตอร์ และทำ Full-text search ได้ในที่เดียว
2.  **Scalability:** รองรับข้อมูลมหาศาล และค้นหาได้รวดเร็วระดับ Milliseconds
3.  **SQL Native:** เราสามารถเขียน Query ที่ซับซ้อนได้โดยใช้คำสั่ง SQL ที่เราคุ้นเคย
