# 🏗️ เรียนรู้ Arona (ตอนที่ 6): สถาปัตยกรรมและโครงสร้างเซิร์ฟเวอร์ (Architecture & Server)

ในตอนนี้เราจะมาดู "กระดูกสันหลัง" ของโปรเจกต์กันครับ ว่า Arona ถูกสร้างขึ้นมาด้วยโครงสร้างแบบไหน และแต่ละส่วนเชื่อมต่อกันอย่างไร

---

## 1. Stack พื้นฐาน (The Engine)
Arona เลือกใช้เทคโนโลยีที่เน้นความเร็วและทันสมัยที่สุดในยุคนี้:
*   **Runtime:** [Bun](https://bun.sh/) - เร็วกว่า Node.js และมีเครื่องมือครบในตัว (Test, Package Manager)
*   **Web Framework:** [ElysiaJS](https://elysiajs.com/) - เฟรมเวิร์กที่เร็วเป็นอันดับต้นๆ และรองรับ TypeScript แบบ 100%
*   **Language:** TypeScript - เพื่อความปลอดภัยของโค้ดและการทำงานร่วมกันเป็นทีม

---

## 2. โครงสร้างโฟลเดอร์ (Project Anatomy)
เมื่อเปิดโปรเจกต์มา สิ่งที่คุณควรสนใจคือ:

*   📂 `src/server.ts`: **"ประตูทางเข้า"** จุดเริ่มต้นของเซิร์ฟเวอร์ที่รวบรวม Module ทุกอย่างเข้าด้วยกัน
*   📂 `src/modules/`: **"ห้องทำงาน"** แบ่งตามฟังก์ชันหลัก (เช่น `ai` สำหรับแชท, `ingest` สำหรับเก็บข้อมูล)
*   📂 `src/libs/`: **"กล่องเครื่องมือ"** ฟังก์ชันช่วยงานต่างๆ ที่แชร์กันใช้ในทุกโมดูล (Database, Auth, Logging)
*   📂 `scripts/`: **"สคริปต์ช่วยงาน"** เช่น การเตรียมฐานข้อมูล หรือการทดสอบค้นหาข้อมูลแบบเร็วๆ

---

## 3. ระบบความปลอดภัย (Authentication & Middleware)
Arona มีระบบป้องกันที่แน่นหนาใน `src/libs/auth.ts`:
1.  **API Keys:** การเข้าใช้งานต้องมี Key ที่ถูกต้อง (เช็คจากไฟล์ `.env` หรือ JSON Config)
2.  **Role-Based Access:** แบ่งระดับผู้ใช้เป็น `admin` (สั่ง Ingest ได้) และ `user` (แชทได้อย่างเดียว)
3.  **Department Scope:** จำกัดการเข้าถึงข้อมูลตามแผนก (เช่น แผนกบัญชี จะไม่เห็นข้อมูลแผนกวิศวกรรม)

---

## 4. การจัดการ Errors และ Logging
*   เราใช้ `src/libs/log.ts` สำหรับบันทึกเหตุการณ์สำคัญ
*   มีการใช้ **OpenTelemetry** ใน `server.ts` เพื่อส่งข้อมูลไปที่ Axiom (ใช้ดูว่าระบบช้าตรงไหน หรือมี Error อะไรเกิดขึ้นใน Production)

---

**สรุป:** Arona ถูกออกแบบมาให้เป็น **Modular Architecture** คือแยกส่วนงานชัดเจน (Ingest แยกจาก AI) เพื่อให้ง่ายต่อการขยายระบบในอนาคต
