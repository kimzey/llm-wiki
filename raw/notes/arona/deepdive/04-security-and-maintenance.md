# Lesson 4: ความปลอดภัย การตั้งค่า และการดูแลรักษา (Security & Maintenance)

ในบทสุดท้ายนี้ เราจะคุยกันเรื่องการทำให้ Arona เป็นระบบที่เสถียร ปลอดภัย และพร้อมสำหรับการใช้งานจริงในระดับองค์กร (Enterprise Ready) โดยเราจะเจาะลึกไปที่ระบบรักษาความปลอดภัย การจัดการคอนฟิก และการดูแลรักษาในระยะยาว

---

## 1. ระบบรักษาความปลอดภัยขั้นสูง (Granular Security)

หัวใจของความปลอดภัยใน Arona ไม่ใช่แค่การเข้ารหัส แต่คือการควบคุมสิทธิ์อย่างละเอียดในระดับบรรทัด (Row-Level Security)

### 1.1 API Key Mapping และ Identity
เราไม่ได้ใช้รหัสผ่านทั่วไป แต่ใช้ **API Key** ที่เก็บในรูปแบบ JSON ใน `.env`:
- **Role Admin:** เข้าถึงได้ทุกอย่าง รวมถึงการสั่ง Index ข้อมูลใหม่
- **Role User:** ถามคำถามได้ตามแผนกที่ตนเองมีสิทธิ์เข้าถึง (เช่น HR, Engineering, Sales)

### 1.2 การคัดกรองข้อมูลตามแผนก (Department Filtering)
ในทุกคำสั่ง SQL ที่ใช้ค้นหาข้อมูล เราจะมีการแนบเงื่อนไขสิทธิ์เข้าไปเสมอ:
```sql
WHERE (department = ANY(${userContext.departments}) OR department = 'all')
```
*ทำไมต้องทำแบบนี้?* เพื่อป้องกันไม่ให้ข้อมูลที่เป็นความลับของแผนกหนึ่งหลุดไปหาอีกแผนกหนึ่งอย่างเด็ดขาด

---

## 2. การจัดการคอนฟิก (Environment Configuration)

การตั้งค่าที่ถูกต้องคือสิ่งที่ทำให้ระบบรันได้อย่างไม่มีปัญหา:

| คีย์สำคัญ | หน้าที่ | ข้อควรระวัง |
|----------|-------|------------|
| `DATABASE_URL` | ต่อเข้า ParadeDB | ห้ามแชร์ URL นี้เด็ดขาด เพราะเข้าถึงข้อมูลดิบได้ทั้งหมด |
| `API_KEYS` | เก็บสิทธิ์พนักงาน | ควรมีการ Update หรือ Rotate Key เป็นระยะๆ |
| `OPENROUTER_API_KEY` | จ่ายค่า AI | ควรตั้ง Limit วงเงินในฝั่ง Provider ไว้ด้วย |

---

## 3. การเฝ้าดูระบบ (Observability)

เพื่อให้เรารู้ว่า Arona สบายดีไหม เราใช้เครื่องมือที่เรียกว่า **Axiom Cloud**:
- **Monitoring:** ดูจำนวนคนใช้งานแบบ Real-time
- **Tracing:** เมื่อ User ถามคำถาม เราจะติดตามได้ว่าคำถามนั้นไป "ช้า" ที่ขั้นตอนไหน (เช่น ช้าที่ Search หรือช้าที่ AI)
- **Token Analytics:** สรุปยอดการใช้ Token รายวันเพื่อให้คุมงบประมาณได้

---

## 4. การดูแลรักษาความรู้ (Knowledge Base Maintenance)

ระบบ RAG จะเก่งเท่ากับข้อมูลที่เราป้อนให้มัน ดังนั้นการดูแลความรู้จึงเป็นงานประจำ:

### 4.1 การ Re-indexing
เมื่อเอกสารนโยบายบริษัทเปลี่ยน เราต้องสั่ง `POST /ingest` ทันที:
- ระบบ Smart Diffing ใน Lesson 2 จะทำงานเพื่ออัพเดทเฉพาะส่วนที่เปลี่ยน
- **ข้อแนะนำ:** ควรทำ Re-index ทุกเช้าก่อนพนักงานเริ่มงาน เพื่อให้ข้อมูลสดใหม่อยู่เสมอ

### 4.2 การล้างข้อมูลที่ล้าสมัย
เอกสารเก่าๆ เช่น "ประกาศปี 2021" อาจทำให้ AI สับสน เราควร:
- ลบเอกสารที่ไม่อยู่ในแผนการใช้งานปัจจุบันทิ้ง
- จัดกลุ่มเอกสารให้ชัดเจน แยกโฟลเดอร์ตามปีหรือตามหมวดหมู่

---

## 5. การติดตั้งและขยายระบบ (Deployment & Scaling)

Arona ถูกออกแบบมาเพื่อความยืดหยุ่น:
- **Docker:** เราใช้ Container ในการรันเพื่อให้มั่นใจว่าสภาพแวดล้อมเหมือนกันทุกที่
- **Stateless Architecture:** คุณสามารถรัน Arona 10 ตัวพร้อมกันได้โดยไม่มีปัญหา เพราะระบบไม่มีการเก็บข้อมูลในเครื่องตัวเอง แต่จะเก็บใน Redis และ Postgres เท่านั้น
- **Scaling Database:** ParadeDB (PostgreSQL) สามารถเพิ่มขนาดได้ง่ายเมื่อข้อมูลมีปริมาณมากขึ้น

---

## 6. คู่มือการแก้ไขปัญหา (Troubleshooting)

| ปัญหา | วิธีเช็ค | วิธีแก้ |
|------|---------|-------|
| AI ตอบไม่ตรง | ดูใน Log ว่า Search เจอ Chunk ไหนมาบ้าง | ปรับหัวข้อ (Header) ใน Markdown ให้ชัดเจนขึ้น |
| ระบบตอบช้ามาก | เช็ค Latency ของฐานข้อมูลใน Axiom | ตรวจสอบ Index ของ Postgres และความเร็วของ Redis |
| Ingest ไม่ผ่าน | เช็คความถูกต้องของไฟล์ Markdown | ตรวจสอบว่าไม่มีอักขระพิเศษที่ทำให้ Chunker พัง |
